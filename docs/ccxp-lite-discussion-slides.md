## Slide 3｜議題一：登入頁版面

### 投影片文字

- 標題：`議題一：登入頁版面`
- 圖片：
  - 現況：`../demo/original/login.png`
  - 原型：`../demo/sidebar/login.png`
- 當前問題：
  - 登入資訊分散
  - 視覺層級不明確
  - 主要任務不夠集中
- 我們的解法：
  - 收斂登入主流程
  - 降低公告干擾
  - 提升欄位可讀性
- 限制與合作：
  - 擴充功能只能重排既有元素
  - 正式版面仍需系統端重整

### 講稿備註

- **當前問題**：登入資訊、公告、驗證碼與其他輔助資訊分散，畫面進入點不夠集中。視覺層級不夠明確，使用者需要自行判斷哪些資訊與當前登入動作最相關。登入本應是低負擔流程，但目前畫面會先帶來閱讀與辨識成本。
- **我們的解法**：重新整理登入頁資訊區塊，把登入欄位、驗證碼與主要操作收斂成更清楚的主流程。調整公告與次要資訊的視覺權重，讓畫面焦點回到登入任務本身。提升欄位、按鈕與提示文字的可讀性，降低首次使用與重複登入的摩擦。
- **我們遇到的限制**：擴充功能只能重排既有頁面元素，無法從系統端重新定義登入流程。若原頁面結構、欄位命名或驗證元件變動，前端調整就需要跟著修正。無法正式接管登入前後的狀態管理、錯誤訊息設計與驗證策略。
- **期待合作方向**：若能從系統端重整登入頁資訊架構，可把登入、公告、說明與例外處理拆成更穩定的正式版面。可一起討論登入頁哪些資訊是每次都應顯示，哪些可延後或轉移，降低主流程干擾。若與驗證機制一併調整，可把登入流程從「填資料加解題」改成更低摩擦的身份確認流程。

## Slide 4｜議題二：無常用功能釘選

### 投影片文字

- 標題：`議題二：無常用功能釘選`
- 圖片：
  - 現況：`../demo/original/main.png`
  - 原型：`../demo/sidebar/main.png`
- 當前問題：
  - 高頻功能沒有固定入口
  - 增加操作路徑
  - 增加 mental load
- 我們的解法：
  - 常用功能收藏
  - 固定區入口
  - 個人化任務捷徑
- 限制與合作：
  - 目前偏好綁瀏覽器
  - 正式收藏需帳號層支援

### 講稿備註

- **當前問題**：高頻功能沒有固定入口，使用者每次都要重新展開選單、重新尋找位置。這會增加不必要的操作路徑，也增加不必要的 mental load。對熟悉系統的使用者來說，真正耗時的常常不是功能本身，而是每次都要重新找到它。
- **我們的解法**：在 ccxpLite 中加入常用功能收藏與固定區，讓使用者能直接進入高頻功能。讓使用者以自己的任務習慣，而不是系統原始分類方式，建立個人常用入口。把常用功能移到側欄前段，降低重複展開與反覆搜尋的成本。
- **我們遇到的限制**：目前收藏資料主要跟隨瀏覽器環境，難以成為正式的跨裝置個人設定。擴充功能只能基於現有頁面推導功能節點，無法直接取得完整、穩定的功能 metadata。若功能名稱、URL 或頁面結構調整，收藏對應可能需要額外維護。
- **期待合作方向**：若能從系統端提供正式的常用功能機制，使用者偏好可以與帳號綁定，而不是綁定單一瀏覽器。可進一步整合近期使用、角色首頁與任務捷徑，形成真正個人化的入口。若有正式 menu API 或功能目錄，收藏、排序與推薦都能更穩定。

## Slide 5｜議題三：功能未分類

### 投影片文字

- 標題：`議題三：功能未分類`
- 圖片：
  - 現況：`../demo/original/main-expanded.png`
  - 原型：`../demo/sidebar/main-expanded.png`
- 當前問題：
  - 分類偏系統邏輯
  - 不對應學生任務
  - 查找成本高
- 我們的解法：
  - 任務導向分類
  - 重組語意群組
  - 以「我要做什麼」找功能
- 限制與合作：
  - 目前分類仍是前端推導
  - 正式資訊架構需系統端梳理

### 講稿備註

- **當前問題**：目前功能分類較接近系統模組或行政邏輯，不一定對應學生的任務認知。使用者在找功能時，常需要先猜它屬於哪個單位或哪個模組，再進一步查找。這會增加查找成本，也增加不必要的 mental load。
- **我們的解法**：ccxpLite 重新以任務導向整理功能群組，例如課務、成績、學籍、繳費、住宿與其他服務。把原本分散的功能入口重組為較容易理解的語意分類。讓使用者用「我要做什麼」來找功能，而不是用「系統把它放在哪裡」來找功能。
- **我們遇到的限制**：擴充功能的分類仍是前端推導結果，不是系統正式資料模型的一部分。若系統功能持續新增、調整或搬移，分類邏輯需要持續人工維護。無法結合角色、權限或學籍狀態做更精準的分類與排序。
- **期待合作方向**：若能與計通一起梳理正式功能目錄，可建立較穩定的任務導向資訊架構。可依學生、教師、行政人員等不同角色提供不同的入口與分類視圖。若有正式的功能 metadata，未來也能支援搜尋、推薦與跨系統整合。

## Slide 6｜議題四：擁擠的 side bar

### 投影片文字

- 標題：`議題四：擁擠的 side bar`
- 圖片：
  - 現況：`../demo/original/select-courses-selected.png`
  - 原型：`../demo/sidebar/select-courses-selected.png`
- 當前問題：
  - 導覽密度高
  - 層級深
  - 搜尋成本高
- 我們的解法：
  - 側欄模式
  - 全螢幕選單模式
  - 分區與留白重整
- 限制與合作：
  - 仍受既有 frame 結構限制
  - 正式導覽模型需系統端重整

### 講稿備註

- **當前問題**：現有導覽區域密度高、層級深，操作路徑與搜尋成本都偏高。同一個側欄同時承載分類、狀態與跳轉功能，但區塊界線與主次關係不夠清楚。這使得使用者即使知道功能存在，也不一定能快速定位。
- **我們的解法**：ccxpLite 將導覽改寫為較清楚的側欄模式與全螢幕選單模式，讓使用者可依情境切換。透過分區、留白、圖示與更清楚的層級，降低單次掃描的視覺負擔。把高頻功能與主要分類放在更前面的位置，縮短定位時間。
- **我們遇到的限制**：擴充功能仍然依附在既有頁面與 frame 結構上，無法完全重做資訊骨架。無法從根本調整系統跨頁一致性，例如不同子系統是否共用同一導覽模型。目前只能先驗證較好的版面可能長什麼樣子，不能直接成為正式設計系統。
- **期待合作方向**：若能從系統端重整導覽，可進一步統一主選單、次選單、當前位置與跨子系統跳轉邏輯。可一起評估除了現有側欄以外，是否有更適合校務系統的版面設計。若能建立正式設計系統，導覽不只會更好看，也會更一致、更可維運。

## Slide 7｜議題五：助理系統

### 投影片文字

- 標題：`議題五：助理系統`
- 當前問題：
  - 手動作業耗時
  - 同步流程有改善空間
  - 具自動化潛力
- 我們的解法：
  - 先辨識高耗時環節
  - 先把流程問題顯性化
  - 再評估自動化價值
- 限制與合作：
  - 使用者端無法承擔正式流程自動化
  - 需要系統端資料流與規則支援

### 講稿備註

- **當前問題**：助理系統相關流程中，部分手動作業非常耗時，使用者回饋顯示重複性高且容易出錯。多個資料來源或資料庫之間的同步機制仍有改善空間，容易造成等待、重工或結果不一致。某些流程其實具備明確規則與重複樣態，理論上存在自動化潛力，但目前仍仰賴人工操作。
- **我們的解法**：ccxpLite 目前主要從使用者端觀察操作流程，協助辨識哪些環節最容易累積時間成本。這份原型的角色比較像是把問題顯性化，讓大家看到哪些流程不只是介面問題，而是流程設計問題。也就是說，我們先證明使用者確實在某些操作上花了不成比例的時間，再討論是否值得進一步自動化。
- **我們遇到的限制**：若沒有系統端的資料流、業務規則與流程節點支援，使用者端無法可靠地做正式自動化。擴充功能無法安全地承擔跨資料來源同步、流程補償、例外處理與權限判斷。即使能做局部快捷化，也很難形成可維運、可追蹤、可審計的正式流程。
- **期待合作方向**：若能一起盤點高重複性作業與資料同步環節，可先找出最適合優化或自動化的流程。可進一步區分哪些問題應由 UI 改善解決，哪些應由後端流程整併或系統整合解決。若系統端能提供正式介面與規則邏輯，未來較有可能做出可靠的流程輔助或半自動化工具。

## Slide 8｜議題六：視覺印象傳統老舊

### 投影片文字

- 標題：`議題六：視覺印象傳統老舊`
- 當前問題：
  - 影響學校印象
  - 影響可靠度印象
  - 影響資訊呈現效率
- 我們的解法：
  - 降低背景干擾
  - 壓低飽和度
  - 重整資訊層級
- 限制與合作：
  - 擴充功能無法建立正式設計規範
  - 需要系統端統一設計語彙

### 講稿備註

- **當前問題**：現行視覺風格容易給人系統老舊的印象，這不只影響美感，也會影響對系統可靠度的主觀感受。畫面中高彩度字體、背景質感與多種舊式裝飾並存，容易造成視覺疲勞與閱讀干擾。UI/UX 上的視覺選擇會直接影響資訊呈現效率，讓使用者更難快速判斷主次。
- **我們的解法**：ccxpLite 透過降低背景干擾、壓低飽和度、重新調整字級與資訊層級，讓內容本身更容易被閱讀。我們沒有大幅改動功能邏輯，而是先用較低成本方式驗證：只要把視覺噪音降下來，整體可讀性就會明顯提升。這也顯示現行介面問題不只是審美問題，而是直接影響操作效率的資訊設計問題。
- **我們遇到的限制**：擴充功能只能疊加在既有畫面之上，無法真正建立正式品牌與介面規範。若缺少系統端統一的設計 token、元件規格與版面規範，視覺改善很難在所有頁面一致落地。子系統之間如果各自獨立演化，使用者仍會面臨風格斷裂與體驗不一致。
- **期待合作方向**：可一起建立一致的前端設計語彙，讓品牌印象、可讀性與操作效率同時提升。若能從系統端建立色彩、字體、元件、狀態與訊息樣式規範，後續維運成本會更低。視覺更新若與導覽、資訊層級與可近用性一起規劃，效果會遠比單純換皮更有價值。

## Slide 9｜議題七：A11y 極度不足

### 投影片文字

- 標題：`議題七：A11y 極度不足`
- 當前問題：
  - HTML semantic 不足
  - ARIA attributes 不足
  - screen reader 支援不足
- 我們的解法：
  - 局部改善焦點與排序
  - 指出依賴滑鼠的流程
  - 讓問題具體化
- 限制與合作：
  - 真正修正需回到原始 HTML 與元件
  - 需建立正式可近用性流程

### 講稿備註

- **當前問題**：現行系統在 HTML semantic、ARIA attributes 與 screen reader 支援方面仍有明顯不足。這不只是少數族群的問題，也會影響鍵盤操作、焦點管理、錯誤提示理解與整體互動穩定性。當表單、導覽與互動元件缺少正確語意時，使用者很難依靠輔助技術順利完成任務。
- **我們的解法**：ccxpLite 可局部改善焦點可見性、資訊排序與部分可讀性，讓既有頁面比較容易使用。在使用者端層面，我們可以先指出哪些互動流程特別依賴滑鼠、哪些資訊缺少清楚標記。這些調整能幫助大家理解可近用性問題的具體樣貌，但它們只能算是補救，不是完整解法。
- **我們遇到的限制**：真正的可近用性需要從原始 HTML、表單結構、互動元件與狀態回饋層面一起調整。擴充功能無法穩定補齊所有語意標記，也無法保證所有子頁面、流程與新功能都持續符合標準。若原始系統沒有內建可近用性測試與開發規範，使用者端修補會一直追著問題跑。
- **期待合作方向**：若能由系統端導入可近用性標準與檢查流程，才能做出穩定且可驗證的改善。可先從登入、主選單、表單、錯誤訊息與常用高頻流程著手，建立基準範例。若能把可近用性納入正式前端開發流程，後續新功能就不需要每次再補救一次。

## Slide 10｜議題八：缺少 RWD 支援

### 投影片文字

- 標題：`議題八：缺少 RWD 支援`
- 當前問題：
  - 現代使用需求已包含 mobile
  - 固定桌面版配置限制大
  - 不同尺寸體驗不一致
- 我們的解法：
  - 先整理資訊架構
  - 驗證窄畫面優先順序
  - 為正式 RWD 做準備
- 限制與合作：
  - 使用者端難以補齊完整 responsive layout
  - 需要系統端重整基礎版面骨架

### 講稿備註

- **當前問題**：現代使用需求已需要考慮 mobile 與不同視窗尺寸，但現行介面明顯以固定桌面版配置為主。當畫面縮小、裝置改變或視窗比例不同時，資訊可能擁擠、截斷，或需要不自然的橫向捲動。即使主要使用情境仍在桌面，缺少響應式設計也會影響不同解析度、分割畫面與輔助工具使用體驗。
- **我們的解法**：ccxpLite 目前優先改善桌面瀏覽情境，先驗證資訊重組、導覽重排與視覺層級整理是否有效。這有助於確認哪些資訊應該優先保留、哪些可折疊、哪些需要在窄畫面重新安排。換句話說，我們先做的是資訊架構層面的整理，讓後續真正做 RWD 時有更清楚的依據。
- **我們遇到的限制**：擴充功能無法替代系統端正式的 responsive layout 設計，也很難穩定處理所有頁面。若原始頁面是以表格、固定寬度與舊式 frame 結構為主，使用者端調整空間有限。在沒有重整基礎版面骨架之前，很難把 mobile 或窄視窗體驗做好。
- **期待合作方向**：若能在系統端評估響應式版面，未來可更完整支援不同裝置與操作情境。可先針對登入、主選單、查詢列表與表單頁面建立 responsive 基準頁型。若與導覽重整、可近用性與設計系統同步規劃，RWD 會更容易落地且更一致。

## Slide 11｜議題九：驗證碼與替代方案評估

### 投影片文字

- 標題：`議題九：驗證碼與替代方案評估`
- 當前問題：
  - 增加登入成本
  - 影響成功率
  - 傳統圖片 CAPTCHA 可被自動化繞過
- 我們目前做法：
  - 本機端模型辨識
  - 自動填入驗證欄位
  - 用原型證明登入摩擦確實存在
- 常見方案：
  - Google reCAPTCHA
  - hCaptcha
  - Cloudflare Turnstile
- 建議結論：`優先評估 Cloudflare Turnstile`

### 講稿備註

- **當前問題**：現行圖片驗證碼增加登入成本，也打斷原本應該很短的登入流程。若驗證碼難讀、載入異常或辨識失敗，使用者會被迫重試。如果 CAPTCHA 的目的在於阻擋自動化行為，那麼純圖片式設計本身也代表其可被自動辨識與繞過。
- **我們的解法**：ccxpLite 目前以本機端模型辨識驗證碼，並自動填入登入或 OAuth 驗證欄位。這個做法的價值不在於把驗證碼「做掉」，而在於證明登入摩擦本身確實是高頻痛點。它也說明現行驗證方式對正常使用者與自動化攻防都不是理想解。
- **我們遇到的限制**：容易受圖片格式、字型、雜訊、尺寸、背景線條與 endpoint 變動影響。依賴輸入欄位名稱、圖片路徑、表單結構或 iframe 結構，穩定性有限。這只能作為使用者端輔助與問題揭示，不能作為校方正式且長期的資安策略。
- **期待合作方向**：若與計通協作，較適合從系統端重新設計驗證流程，而不是持續強化圖片 CAPTCHA。可依登入風險、異常來源與失敗次數調整驗證強度，減少所有使用者都必須被同樣對待。驗證設計若能兼顧低摩擦、可近用性與正式伺服器端驗證，整體使用體驗會更穩定。
- **常見替代方案**：Google reCAPTCHA 是市面上最常見的方案之一，提供風險分析、分數、企業防護與多種整合能力，但整體定位更偏大型通用風控平台。hCaptcha 是常見的 CAPTCHA 替代方案，強調隱私與企業防護能力，也可做風險評分與自適應驗證。Cloudflare Turnstile 則被官方定位為 CAPTCHA alternative，可嵌入任何網站，且不需要把整站流量導入 Cloudflare。
- **為什麼我們優先建議 Cloudflare Turnstile**：官方提供 Managed、Non-Interactive、Invisible 三種 widget mode；其中 Managed mode 為官方推薦模式，會依風險自動決定是否需要額外互動，通常不需要辨識圖片或文字。這較符合登入頁情境，因為我們的重點是降低登入阻力，而不是把登入頁變成高干擾風控頁面。整合方式也相對單純，前端取得 token，後端再用 `siteverify` 驗證即可。根據 Cloudflare 官方文件，Turnstile 可嵌入任何網站，不需把網站流量全部接到 Cloudflare。相較傳統圖片驗證碼，也更有機會降低閱讀負擔、輸入負擔與辨識錯誤。
- **為什麼不是先選其他方案**：不是先選傳統圖片 CAPTCHA，因為這正是目前問題來源，既增加登入成本，也已被證明可以被自動化繞過。不是優先選 reCAPTCHA，因為雖然它成熟且功能強，但若目標是先改善校務系統登入體驗，它並沒有比 Turnstile 更明確的低摩擦優勢。不是優先選 hCaptcha，因為它也是可行選項，也有隱私與企業防護主張，但若以導入簡潔度、登入頁干擾程度與目前原型要驗證的方向來看，Turnstile 更貼近這次需求。
- **導入時需注意**：Invisible mode 可能需要依 Cloudflare 文件在隱私政策中揭露 Turnstile Privacy Addendum。仍需評估校方資安、個資、採購、網路連線、可用性與備援策略。應保留無障礙與替代驗證流程，避免第三方服務異常時阻斷正常使用者。

## Slide 13｜結語

### 投影片文字

- 標題：`結語`
- 結論：`ccxpLite 證明了使用者端改善可以先帶來可觀效益，但要走向穩定、安全、可維運的方案，仍需要與計通協作。`

### 講稿備註

- ccxpLite 的價值在於以低成本方式呈現「使用者端改善可以帶來什麼」。
- 目前的原型已經證明，很多痛點並不是非得大改後端才能看見改善，資訊架構、導覽與視覺層級本身就能明顯影響使用效率。
- 但若要把這些改善正式落地，後續仍需要計通中心一起評估系統架構、維運限制、資安規範與整合方式，才能把這些觀察轉化為更穩定、安全且可維運的校務資訊系統改善方案。
